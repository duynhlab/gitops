# gitops

GitOps source of truth for the duynhlab platform. Owned by **DevOps/SRE**;
developers never push here directly — changes arrive as pull requests from
[Backstage](https://github.com/duynhlab/backstage) self-service templates or
from service CI (staging auto-deploy). Flux on each cluster applies what is on
`main`.

## Environments

Four logical environments across **two Kind clusters**:

| Env | Cluster | Namespace | Update path | Review |
|-----|---------|-----------|-------------|--------|
| staging | `kind-dev` | `<svc>-staging` | CI image-automation → commit `main` | none (auto) |
| beta | `kind-prod` | `<svc>-beta` | Backstage PR | required |
| prod-us | `kind-prod` | `<svc>-prod-us` | Backstage PR | required |
| prod-eu | `kind-prod` | `<svc>-prod-eu` | Backstage "Promote Image" → image-automation PR | required |

A service is **onboarded to staging only**, then promoted to higher
environments on demand via the "Enable Environment" template. `kind-dev` Flux
syncs `clusters/staging`; `kind-prod` Flux syncs `clusters/prod`.

## Layout

```
clusters/
  staging/    sources.yaml · apps.yaml (→ ../../apps/staging) · image-automation-<svc>.yaml
  prod/       sources.yaml (+ apps-<env>.yaml added by Enable Environment)
apps/
  base/<svc>/values-base.yaml          # env-invariant Helm values (mop chart)
  staging/<svc>/                       # namespace · helmrelease · externalsecret
                                       # values-staging.yaml · kustomization (configMapGenerator)
  beta|prod-us|prod-eu/<svc>/          # created by Enable Environment
service-registry/<svc>.yaml            # per-service metadata (onboard writes one file)
catalog/<svc>.yaml                     # Backstage entity (discovered, never applied by Flux)
```

## How values flow into a release

Each environment's `HelmRelease` reads **two** ConfigMaps via `valuesFrom`,
generated by kustomize `configMapGenerator` from files in git:

1. `values-base.yaml` — env-invariant (image repo, service, probes, resources)
2. `values-<env>.yaml` — the per-env desired state (image tag, replicas, the
   full env-var list, secret refs)

Later wins, so `values-<env>` overrides base; maps deep-merge (image.repository
from base + image.tag from the env file), the env list is taken wholesale from
the env file. `disableNameSuffixHash: true` keeps ConfigMap names stable for the
`valuesFrom` refs; Flux still reconciles when content changes.

> Cross-directory file ref (`../../base/checkout/values-base.yaml`) is resolved
> by Flux kustomize-controller (`LoadRestrictionsNone`). Build locally with
> `kubectl kustomize --load-restrictor LoadRestrictionsNone apps/<env>/<svc>`.

## Image automation

`clusters/<cluster>/image-automation-<svc>.yaml` holds an ImageRepository +
ImagePolicy + ImageUpdateAutomation. The `image.tag` line in each
`values-<env>.yaml` carries a `# {"$imagepolicy": "flux-system:<svc>-<env>:tag"}`
marker; automation rewrites it in place.

- **staging** — ImagePolicy takes the newest CI tag (`main-<ts>-<sha>`, sorted
  numerically) and commits straight to `main` (auto).
- **beta / prod-us / prod-eu** — automation pushes to a `flux-image-<env>`
  branch and a GitHub Action opens a PR (reviewed). `prod-eu` only matches
  `eu-*` tags produced by the "Promote Image" flow.

Automation authenticates to git with a **GitHub App** (scoped to this repo),
not a personal token.

## Secrets

Each service/env carries an `ExternalSecret` + a per-service `SecretStore`
(External Secrets Operator). On Kind the store uses the ESO **`fake`** provider
(static demo value) so onboarding/enabling only ever ADDS files — never edits a
shared store. A real cluster drops the per-service store and points the
ExternalSecret at a shared `ClusterSecretStore` backed by Vault / AWS Secrets
Manager, untouched by the service.

## Guard rails

- `CODEOWNERS`: `apps/beta|prod-us|prod-eu/**`, `apps/base/**`, `clusters/**`
  require `@duynhne`; `apps/staging/**` is intentionally unowned so staging-only
  PRs (onboarding, CI, env-var tweaks) merge freely.
- Branch protection uses **require review from Code Owners** (not a blanket
  count) so the split above holds.
- Images are immutable; promotion moves a tag through env files / regions,
  never rebuilds.
